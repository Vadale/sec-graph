> **Status: complete (2026-07-19).** This document's tables are generated by

> the claims below are written **after** the numbers exist, reporting what the data shows — not a

> pre-registration, locked by a Fable 5 max design review). Ground-truth labels were drafted blind
> (SARIF + source only, no sec-graph output), citation-backed, and independently audited.

## What this measures (and what it does not)

sec-graph does **not** find vulnerabilities — CodeQL does. The hypothesis is narrower and, we think,
more useful: **given the same SAST findings, does sec-graph's view (enrichment layers + minimal
hash-verified source→sink slices + unguarded-first prioritization) make _triage_ by a small local LLM
more accurate, cheaper, and better-prioritized than triaging the raw SAST output?**

- **Arm A (control, steelman):** the LLM sees the rendered CodeQL finding **plus the full numbered
  text of every file in its data-flow**. A's evidence is a strict *superset* of the raw material
  behind B — a B win cannot be "A lacked information."
- **Arm B (treatment):** the LLM sees exactly the sec-graph **MCP payload** (enrichment layers, the
  auth/unguarded verdict, and the minimal ±3-line hash-verified windows).
- **Arm C (ablation):** B's slices with the enrichment stripped — isolates *minimal slices* from
  *enrichment*.

Everything else (model `gemma4:e4b-it-qat`, decoding, prompt skeleton, output schema, the 18-class
taxonomy, the finding set) is held identical. One call per finding; **intent-to-treat** denominators
(a bind/parse/ctx failure scores that arm *wrong*, never a silent drop).

**Corpus** (66 findings; per-repo N is small — a documented limitation, single-digit cells reported as
counts): **PyGoat** (40, the *development* corpus — rules/dicts/prompt were tuned on it, so its results
are split out), **Vulpy** (17; ships `bad/`+`good/` for false-positive mass) and **Vulnerable-Flask-App**
(9), both **unseen**. Full survey (incl. rejected repos) in `corpus/manifest.json`.

**Every claim is scoped** to this model, these repos, Python, and triage of an existing SAST's findings
— never detection, humans, other models/languages, or production codebases.

## Reading of the results

The honest bottom line is **efficiency, not accuracy** — and the self-audit caught a real bug in our
own tool. In order of how well the data supports each claim:

**1. The robust result is cost.** Arm B reached the same-or-better triage quality as the whole-file
control while reading **~1/7th the prompt tokens** (46,021 vs 319,765 pooled) and finishing in **~half
the wall-time** (271 s vs 480 s). On the large-file repo the gap is starkest: PyGoat median prompt
tokens **689 (B) vs 8,195 (A)** — a 12× reduction, and whole files there barely fit a 16 K local
window at all. For a small local model, this is the defensible value: the minimal hash-verified slice
is what makes per-finding triage cheap enough to run.

**2. Real/FP accuracy: a small, underpowered, repo-dependent edge — not a win we can assert.**
Pooled Arm B 91% vs Arm A 85% (**Δ +6 pts, 95% CI [0, +14], exact McNemar p = 0.22** on 6 discordant
pairs — *not* significant). And it is **mixed per repo**: B wins big on Vulnerable-Flask-App
(44% → 100%) but **A wins on Vulpy** (71% vs 65%). The mechanism is interpretable and honest: whole-file
context biases toward "real" on a vulnerable file (A missed VFA's 5 FPs), while a focused slice keeps
the model on the flagged flow — *unless* the false-positive signal is global (Vulpy's FPs are attack-
tool scripts whose nature is only visible in the whole file, so A's context helped and B's slice hurt).
**So the value of slice-vs-whole-file depends on whether the FP-distinguishing evidence is local or
global.** With N = 66 and 11 FPs, we report this as a signal, not a result.

**3. The guard verdict currently HURTS — and that is the finding.** Arm A (83%) beat Arm B (71%) on
auth-guard accuracy, because B inherits sec-graph's guard errors. The **tool self-audit** localizes it:
sec-graph's "unguarded" verdict is right only **15/27 (56%)** of the time, while "guarded" is **14/14
(100%)** — i.e. it **never falsely claims guarded** (ADR-010's asymmetry holds) but **over-reports
unguarded**. Root cause, traced by inspecting all 12 false-`unguarded`s (an earlier draft of this file
mis-attributed it to inline `if request.user.is_authenticated:` — that is *wrong*; sec-graph handles
that gate correctly, verified). The 12 break down as: **(a) custom barrier names the default rule pack
doesn't list** — PyGoat's `@authentication_decorator` (7), Vulnerable-Flask-App's `verify_jwt()` and
Vulpy's `libuser.login()` (3); the barrier list in `rules/labels.yml` is user-configurable *precisely*
because the tool cannot know every app's auth-function names by default; and **(b) interprocedural
guards** — the sink's enclosing function is unguarded but a *caller* gates it (2: the cross-file
path-injection, the `password_change` helper), which the ingest-time guard analysis (intraprocedural)
does not see. So this is an honest **limitation**, not a one-line bug: it's *configuration* (add your
app's barrier names) + a real future improvement (interprocedural guard propagation in the enrich path).
Adding those specific names to the default would **overfit this corpus**, so we don't. It also degrades
O3's unguarded-first tiering (below).

**4. Prioritization (O3 vs O2): no win on this corpus.** Against a severity-sorted SARIF baseline (O2),
sec-graph's ordering (O3) did **not** do better — both hit the first true-critical at rank 1 on 2/3
repos, and O2's precision@10 was competitive or better (PyGoat 10/10 vs 7/10). The unguarded-first
signal that is supposed to differentiate O3 is degraded by the same false-unguarded limitation in (3). Honest
verdict: on this small, mostly-critical corpus, a `jq` severity sort is a strong baseline we did not beat.

**5. Class accuracy is confounded — we do not claim it.** Arm B "won" class (98% vs 49%), but B is
handed the CWE in its payload while A must infer the class from code, so this metric is near-ceiling for
B *by construction* (flagged in PROTOCOL §4). The fair test is the blind-class condition — future work.

**6. Fully deterministic.** run0 vs run1 agreed on **198/198 (100%)** verdicts (temp 0 + fixed seed).

**Strongest claim we will make:** *for a small local model (Gemma 3n E4B) triaging the same 66 CodeQL
findings on three deliberately-vulnerable Python apps, sec-graph's minimal MCP slices matched whole-file
triage on real/FP accuracy (Δ +6 pts, not significant) while cutting prompt tokens ~7× and wall-time
~2×; its current guard verdict and prioritization did not improve over the raw baseline, and the
self-audit surfaced a false-`unguarded` limitation (custom barrier names + interprocedural guards),
though never a false-`guarded`.* Scoped to this model, these repos, and Python.

## When it helps / when it does not

**Helps triage:**
- **Many findings in large files** — whole-file triage (Arm A) costs an order of magnitude more prompt
  tokens per finding and does not fit a small local context window; slices (Arm B) do.
- **Cross-file / multi-hop flows** — the hop windows answer "which file do I even open"; the raw
  workflow must load every file on the flow.
- **Small-context local models** — slices fit where whole files do not; fewer prompt tokens ≈
  proportionally faster prefill on local hardware.
- **Re-triage during fix loops** — hash-verified staleness flags on each window; the raw workflow has
  no drift detection.

**Does not help, or can hurt:**
- **Tiny single-file apps** — whole file ≈ slice; the token/latency delta collapses to ~0.
- **Config/template-dependent classes** (CSRF middleware, cookie flags, template XSS) — a ±3-line
  window can omit the decisive evidence a whole-file view has; Arm A may win these cells.
- **Non-Python sinks** — the guard verdict is honestly `unknown` (no glow, no ranking signal beyond
  severity).
- **Lexical, not semantic, enrichment** — misses renamed/indirect sensitive data and can noise-tag;
  confidence is capped at medium by design.
- **Upstream-recall-bound** — anything CodeQL did not emit exists in neither arm; sec-graph adds no
  findings.
- **Model reasoning errors survive good evidence** — better context reduces but does not eliminate
  them.
- **FastAPI `Depends()` auth** reads `unknown`/`unguarded` (a documented under-claim, ADR-015) → it
  over-prioritizes FastAPI routes.
- **The ingest-time guard verdict over-reports `unguarded`** (56% precision here; never a false
  `guarded`, 100%) when the app uses a **custom auth decorator/callable** not in the default
  `rules/labels.yml` barrier list (add your app's names — that is what the list is for) or when the
  **guard is in a calling function** (the enrich-time analysis is intraprocedural). It handles inline
  `if request.user.is_authenticated:` correctly. This hurt Arm B's guard accuracy and the O3 ordering.
  Improvement = interprocedural guard propagation in the enrich path (future work); we do **not** add
  the corpus's specific barrier names to the default (that would overfit this benchmark).

## Reproduce

```
python benchmark/fetch.py      # clone corpus @ pinned commits, secgraph analyze (uses the committed SARIFs)
python benchmark/run.py        # the A/B/C sweep -> results/verdicts.json   (local Ollama; ~1-2 h)
python benchmark/report.py     # score vs truth.json -> the tables above
```
